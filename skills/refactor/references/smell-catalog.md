# Code Smell Catalog — Diagnosis, Not Definitions

Claude already knows what a long method or a god class is. This file exists only for the parts that take experience: the detection threshold, and the non-obvious trap in the "obvious" fix.

| Smell                  | Detection cue                                                                                 | Fix                                                                                                           | Non-obvious trap                                                                                                                                                                                                    |
| ---------------------- | --------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Long Method            | Needs a comment per section to follow, or you're scrolling to read it.                        | Extract Method at natural seams — blocks with a single purpose, or blocks preceded by an explanatory comment. | Extracting at the wrong seam (splitting by line count, not by concern) just relocates the mess into two functions instead of one.                                                                                   |
| God Class/Module       | 20+ methods spanning unrelated domains; imported by nearly everything.                        | Extract Class, grouped by what data each group touches.                                                       | Splitting by _method name similarity_ instead of _shared data_ produces classes that are still coupled through the data they secretly share.                                                                        |
| Duplicated Code        | Same logic copy-pasted 3+ places.                                                             | Extract the shared logic.                                                                                     | Two occurrences is not yet a signal — wait for the third before abstracting (see Critical Rules). Two blocks that _look_ similar but express different domain concepts should stay separate even at 3+ occurrences. |
| Long Parameter List    | 5+ params; callers pass `null` for ones they don't care about; params always travel together. | Introduce a Parameter Object.                                                                                 | If the grouped params belong to _different concerns_, the real fix is splitting the function, not just bundling the params — bundling can paper over a function doing two jobs.                                     |
| Primitive Obsession    | Domain concepts as raw string/number/bool; validation scattered at every callsite.            | Value Object or branded type, validated at construction.                                                      | Only worth it if the invariant is checked in more than one place — a single-callsite validation doesn't need a type, just a check.                                                                                  |
| Feature Envy           | A method in A mostly reads fields from B.                                                     | Move the method to B.                                                                                         | If A's method also needs _A's own_ state, moving it to B creates the reverse problem (B now reaching into A) — sometimes the right fix is moving data, not the method.                                              |
| Shotgun Surgery        | One behavior change requires edits across many unrelated files.                               | Centralize the behavior into one owner (class/module/function).                                               | Centralizing without dependency inversion just creates a new single point that every caller must import — confirm the new owner doesn't become a god module itself.                                                 |
| Data Clumps            | Same 3-4 fields always travel together across functions/classes/DB.                           | Group into a named object (`Address`).                                                                        | Worth doing only if the group has shared _behavior_ or _validation_ to attach — if it's purely cosmetic grouping, it's ceremony.                                                                                    |
| Switch/if-else on Type | Multiple switches branch on the same type field; new type means editing every switch.         | Polymorphism if you own the type; dispatch table (map) if you don't.                                          | Don't reach for this until a _second_ switch on the same field appears — a single switch is often clearer than a one-class-per-variant split.                                                                       |
| Magic Numbers/Strings  | Literal value with no explanation (`if status === 2`).                                        | Named constant expressing the concept, not the value.                                                         | Naming the constant after the _current_ meaning when the value is really an arbitrary tunable (timeout, threshold) hides that it should be configurable, not hardcoded with a nicer name.                           |
| Dead Code              | Unused imports, unreachable branches, commented-out blocks, uncalled functions.               | Delete it; git history is the backup.                                                                         | Grep callsites first — "unused" via static read can miss dynamic dispatch (reflection, string-based routing, plugin registries).                                                                                    |
| Nested Conditionals    | 4+ indent levels; else branches as long as the then branches.                                 | Guard clauses (early return for edge/error cases) so the happy path runs straight down.                       | Guard-clause-ing a condition that's actually part of the business logic (not just an edge case) buries an important branch as if it were boilerplate.                                                               |
