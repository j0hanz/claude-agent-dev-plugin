# Structural Refactoring Patterns — When to Apply, When Not To

Claude already knows how to implement Strategy, Builder, Command, etc. This file is not a tutorial — it's the judgment call: which symptom actually warrants which pattern, and the non-obvious way each one bites back if misapplied.

| Pattern                     | Apply when                                                                                                                                         | Don't apply when                                                                                                                                                                         | Non-obvious caution                                                                                                                                                                                                   |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Strategy**                | A function has a large if/switch on a type field, each branch is similar-shaped, and new variants are expected over time.                          | Only 2-3 branches exist and no new variant is foreseeable — a switch is more readable than a map of one-line lambdas.                                                                    | The payoff is zero-edit extensibility (new variant = new map entry). If callers still need to enumerate all variants elsewhere (e.g., a UI dropdown), the map doesn't remove that coupling — it just moves it.        |
| **Null Object**             | Code is full of `if (x !== null)` guards and the absent case has one obvious default behavior.                                                     | The "absent" case needs different handling at different callsites — collapsing it into one default silently picks a behavior for all of them.                                            | Replacing `null` with a sentinel object can hide real "missing data" bugs that the null check used to catch loudly. Only safe when absence is a normal, expected state, not an error condition.                       |
| **Builder**                 | An object has many optional fields, constructor calls pass `null`/`undefined` for args the caller doesn't care about, or test setup is unreadable. | Object has 2-3 fields, all required — a builder adds ceremony with no payoff.                                                                                                            | If the language has named/keyword arguments or object-literal construction, that's usually enough; reach for Builder only when there's required _ordering_ or _validation_ across steps.                              |
| **Command**                 | Operations need to be queued, logged, undone, or retried, and the logic is currently inline in a handler.                                          | The operation is fire-and-forget with no undo/retry/audit requirement — wrapping it in a class just to "be clean" is ceremony with no payoff.                                            | Undo logic is the part people skip — a Command without a real `undo()` is just an indirection layer, not the pattern.                                                                                                 |
| **Chain of Responsibility** | A sequence of independent checks/filters, each of which can short-circuit (validation pipelines, middleware, permission checks).                   | Checks must run in a fixed order with shared mutable state, or every check must always run regardless of earlier results — a plain array `.map().filter()` is simpler than a true chain. | Order-dependent rules (rule B only makes sense if rule A passed) reintroduce the coupling the pattern was supposed to remove. Keep rules independent or don't use this.                                               |
| **Observer / Event Bus**    | One action must trigger several genuinely unrelated reactions (email, inventory, analytics) and the direct-call coupling is already causing pain.  | Used just to avoid one import, or there are only 1-2 listeners — the indirection costs more than the coupling it removes.                                                                | The #1 real-world failure mode: emitters and listeners drift apart over time and the causal chain becomes untraceable ("magic" behavior). Only justified when the decoupling is genuinely valuable, not as a default. |

**Cross-cutting rule:** every row above trades direct coupling for indirection. Indirection has a real cost (harder to trace, harder to step through in a debugger). Apply a pattern only when the "Apply when" column's symptom is already causing measurable pain — never apply preemptively for code that might grow that way.
